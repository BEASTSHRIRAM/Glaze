# Requirements Document

## Introduction

Glaze is a full-stack application that enables semantic search over Google Drive files using AI-powered embeddings. The system consists of a Chrome Extension frontend and a FastAPI backend that integrates with Google Drive API, Google Gemini Embedding Model, and Qdrant vector database. Users can connect their Google Drive account, index their files, and perform natural language queries to find relevant documents based on meaning rather than keyword matching.

## Glossary

- **Glaze_System**: The complete application including Chrome Extension, Backend API, and supporting services
- **Chrome_Extension**: The browser-based user interface for search and authentication
- **Backend_API**: The FastAPI server handling authentication, indexing, and search operations
- **Embedding_Service**: The component that generates vector embeddings using Google Gemini Embedding Model
- **Indexing_Service**: The component that processes Drive files and stores embeddings
- **Qdrant_Database**: The vector database storing file embeddings for similarity search
- **Metadata_Database**: The SQLite database storing file metadata (fileId, name, link, mimeType)
- **OAuth_Handler**: The component managing Google OAuth 2.0 authentication flow
- **File_Processor**: The component that extracts text content from various file formats
- **Search_Engine**: The component that performs semantic similarity search
- **Chunk**: A segment of file content (500-1000 tokens) used for embedding generation
- **Embedding**: A vector representation of text content generated by Gemini model
- **Vector_Search**: Similarity search performed on embeddings in Qdrant

## Requirements

### Requirement 1: User Authentication

**User Story:** As a user, I want to authenticate with my Google account, so that the system can access my Google Drive files securely.

#### Acceptance Criteria

1. WHEN a user initiates authentication, THE OAuth_Handler SHALL redirect the user to Google OAuth 2.0 consent screen
2. WHEN Google returns an authorization code, THE OAuth_Handler SHALL exchange it for an access token
3. THE OAuth_Handler SHALL store the access token securely for subsequent API calls
4. WHEN an access token expires, THE OAuth_Handler SHALL refresh the token using the refresh token
5. IF authentication fails, THEN THE OAuth_Handler SHALL return a descriptive error message to the user

### Requirement 2: Google Drive File Retrieval

**User Story:** As a user, I want the system to fetch my Google Drive files, so that they can be indexed for search.

#### Acceptance Criteria

1. WHEN a user requests file indexing, THE Backend_API SHALL retrieve the list of files from Google Drive API
2. THE Backend_API SHALL fetch file metadata including fileId, name, mimeType, and webViewLink
3. THE Backend_API SHALL filter files by supported mimeTypes (application/pdf, application/vnd.google-apps.document, text/plain, application/vnd.ms-powerpoint, application/vnd.openxmlformats-officedocument.presentationml.presentation)
4. IF the Google Drive API returns an error, THEN THE Backend_API SHALL log the error and return an error response
5. THE Backend_API SHALL handle pagination for users with more than 100 files

### Requirement 3: File Content Extraction

**User Story:** As a developer, I want to extract text content from various file formats, so that embeddings can be generated for search.

#### Acceptance Criteria

1. WHEN a PDF file is processed, THE File_Processor SHALL extract text content using a PDF parsing library
2. WHEN a DOCX file is processed, THE File_Processor SHALL extract text content preserving document structure
3. WHEN a TXT file is processed, THE File_Processor SHALL read the text content directly
4. WHEN a PPT or PPTX file is processed, THE File_Processor SHALL extract text from slides
5. IF a file cannot be processed, THEN THE File_Processor SHALL log the error and skip the file
6. THE File_Processor SHALL split extracted content into Chunks of 500-1000 tokens each

### Requirement 4: Embedding Generation

**User Story:** As a developer, I want to generate vector embeddings for file content, so that semantic search can be performed.

#### Acceptance Criteria

1. WHEN text content is provided, THE Embedding_Service SHALL generate a vector embedding using Google Gemini Embedding Model (multimodal embedding 2)
2. THE Embedding_Service SHALL process embeddings in batches to optimize API usage
3. WHEN the Gemini API rate limit is reached, THE Embedding_Service SHALL implement exponential backoff retry logic
4. IF embedding generation fails for a chunk, THEN THE Embedding_Service SHALL log the error and continue with remaining chunks
5. THE Embedding_Service SHALL return embeddings as numerical vectors compatible with Qdrant storage format

### Requirement 5: Vector Database Storage

**User Story:** As a developer, I want to store embeddings in Qdrant, so that similarity search can be performed efficiently.

#### Acceptance Criteria

1. WHEN the system initializes, THE Indexing_Service SHALL create a Qdrant collection named "drive_files" if it does not exist
2. WHEN an embedding is generated, THE Indexing_Service SHALL store it in Qdrant with a unique chunk identifier
3. THE Indexing_Service SHALL store payload data including file_id, file_name, chunk_text, and mime_type with each vector
4. THE Indexing_Service SHALL use appropriate vector dimensions matching Gemini embedding output (768 dimensions)
5. IF Qdrant storage fails, THEN THE Indexing_Service SHALL return an error and halt the indexing process for that file

### Requirement 6: Metadata Database Storage

**User Story:** As a developer, I want to store file metadata in SQLite, so that file information can be retrieved quickly after search.

#### Acceptance Criteria

1. WHEN the system initializes, THE Backend_API SHALL create a SQLite database with a "files" table if it does not exist
2. THE Metadata_Database SHALL store columns: id, file_id, file_name, mime_type, and link
3. WHEN a file is indexed, THE Backend_API SHALL insert or update the file metadata in the Metadata_Database
4. THE Backend_API SHALL ensure file_id is unique in the Metadata_Database
5. WHEN search results are retrieved, THE Backend_API SHALL query the Metadata_Database using file_id to fetch complete metadata

### Requirement 7: Incremental Indexing

**User Story:** As a user, I want the system to avoid re-indexing already processed files, so that indexing is faster and more efficient.

#### Acceptance Criteria

1. THE Indexing_Service SHALL maintain a record of indexed file identifiers
2. WHEN indexing is triggered, THE Indexing_Service SHALL check if a file_id already exists in the Metadata_Database
3. WHEN a file has already been indexed, THE Indexing_Service SHALL skip processing that file
4. WHEN a file has been modified since last indexing, THE Indexing_Service SHALL re-process and update the embeddings
5. THE Indexing_Service SHALL provide a force re-index option to override incremental behavior

### Requirement 8: Semantic Search Query Processing

**User Story:** As a user, I want to search my Drive files using natural language queries, so that I can find relevant documents based on meaning.

#### Acceptance Criteria

1. WHEN a user submits a search query, THE Search_Engine SHALL generate an embedding for the query text using the Embedding_Service
2. THE Search_Engine SHALL perform vector similarity search in Qdrant_Database using the query embedding
3. THE Search_Engine SHALL retrieve the top 10 most similar results by default
4. THE Search_Engine SHALL fetch complete file metadata from Metadata_Database for each result
5. THE Search_Engine SHALL return results including file_name, mime_type, link, chunk_text, and similarity score
6. THE Search_Engine SHALL complete search operations within 2 seconds for typical queries

### Requirement 9: Chrome Extension User Interface

**User Story:** As a user, I want a clean search interface in my browser, so that I can easily search my Drive files.

#### Acceptance Criteria

1. THE Chrome_Extension SHALL provide a popup interface with a search input box
2. WHEN a user enters a query and submits, THE Chrome_Extension SHALL send the query to the Backend_API
3. WHILE a search is in progress, THE Chrome_Extension SHALL display a loading spinner
4. WHEN search results are received, THE Chrome_Extension SHALL display them in a list format
5. THE Chrome_Extension SHALL display file name, file type icon, and a content snippet for each result
6. THE Chrome_Extension SHALL provide an "Open in Drive" button for each result that opens the file in Google Drive
7. IF no results are found, THEN THE Chrome_Extension SHALL display a "No results found" message
8. IF an error occurs, THEN THE Chrome_Extension SHALL display a user-friendly error message

### Requirement 10: Chrome Extension Manifest and Permissions

**User Story:** As a developer, I want the Chrome Extension to have proper permissions, so that it can function correctly and securely.

#### Acceptance Criteria

1. THE Chrome_Extension SHALL use Manifest V3 format
2. THE Chrome_Extension SHALL request "storage" permission for storing user preferences
3. THE Chrome_Extension SHALL request "identity" permission for Google authentication
4. THE Chrome_Extension SHALL declare the Backend_API endpoint in host_permissions
5. THE Chrome_Extension SHALL implement Content Security Policy to prevent XSS attacks

### Requirement 11: Docker Infrastructure Setup

**User Story:** As a developer, I want to run Qdrant using Docker, so that the vector database is easy to set up and manage.

#### Acceptance Criteria

1. THE Glaze_System SHALL provide a docker-compose.yml file for Qdrant deployment
2. THE docker-compose.yml SHALL configure Qdrant to run on port 6333
3. THE docker-compose.yml SHALL configure persistent volume storage for Qdrant data
4. WHEN docker-compose is executed, THE Qdrant_Database SHALL start and be accessible at localhost:6333
5. THE Backend_API SHALL connect to Qdrant using the configured host and port

### Requirement 12: Backend API Endpoints

**User Story:** As a developer, I want well-defined API endpoints, so that the Chrome Extension can communicate with the backend.

#### Acceptance Criteria

1. THE Backend_API SHALL provide a POST /auth/google endpoint that initiates OAuth login
2. THE Backend_API SHALL provide a GET /auth/callback endpoint that handles Google OAuth callback
3. THE Backend_API SHALL provide a GET /drive/files endpoint that returns the list of user's Drive files
4. THE Backend_API SHALL provide a POST /index endpoint that triggers file indexing
5. THE Backend_API SHALL provide a POST /search endpoint that accepts a query string and returns search results
6. THE Backend_API SHALL return responses in JSON format with appropriate HTTP status codes
7. THE Backend_API SHALL implement CORS headers to allow requests from the Chrome Extension

### Requirement 13: Error Handling and Logging

**User Story:** As a developer, I want comprehensive error handling and logging, so that I can debug issues and ensure system reliability.

#### Acceptance Criteria

1. WHEN an error occurs in any component, THE Glaze_System SHALL log the error with timestamp, component name, and error details
2. THE Backend_API SHALL return structured error responses with error codes and messages
3. IF the Gemini API is unavailable, THEN THE Embedding_Service SHALL return a service unavailable error
4. IF Qdrant is unreachable, THEN THE Indexing_Service SHALL return a database connection error
5. THE Backend_API SHALL log all API requests with method, endpoint, and response status

### Requirement 14: Configuration Management

**User Story:** As a developer, I want centralized configuration, so that API keys and settings can be managed easily.

#### Acceptance Criteria

1. THE Backend_API SHALL load configuration from environment variables
2. THE Backend_API SHALL require GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, and GEMINI_API_KEY environment variables
3. THE Backend_API SHALL provide default values for optional configuration like QDRANT_HOST and QDRANT_PORT
4. IF required environment variables are missing, THEN THE Backend_API SHALL fail to start with a descriptive error message
5. THE Backend_API SHALL not log or expose sensitive configuration values

### Requirement 15: Parser and Serializer Round-Trip Testing

**User Story:** As a developer, I want to ensure data integrity in serialization, so that embeddings and metadata are stored and retrieved correctly.

#### Acceptance Criteria

1. THE Glaze_System SHALL provide a serializer for converting embedding vectors to Qdrant storage format
2. THE Glaze_System SHALL provide a deserializer for retrieving embedding vectors from Qdrant
3. FOR ALL valid embedding vectors, serializing then deserializing SHALL produce an equivalent vector (round-trip property)
4. THE Glaze_System SHALL provide a serializer for converting file metadata to SQLite storage format
5. FOR ALL valid metadata objects, serializing then deserializing SHALL produce an equivalent object (round-trip property)
